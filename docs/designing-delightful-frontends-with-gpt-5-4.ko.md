# Designing Delightful Frontends with GPT-5.4

> GPT-5.4로 더 매력적인 프론트엔드 만들기

**원문 출처:** [https://developers.openai.com/blog/designing-delightful-frontends-with-gpt-5-4](https://developers.openai.com/blog/designing-delightful-frontends-with-gpt-5-4)

---

GPT-5.4는 이전 모델들보다 더 뛰어난 웹 개발자이며, 더 시각적으로 매력적이고 더 과감한 프론트엔드를 생성할 수 있습니다. 특히 OpenAI는 UI 역량과 이미지 활용 능력 향상에 초점을 맞춰 GPT-5.4를 학습시켰습니다. 적절한 가이드를 제공하면, 이 모델은 섬세한 디테일과 잘 다듬어진 인터랙션, 아름다운 이미지를 갖춘 실서비스 수준의 프론트엔드를 만들어낼 수 있습니다.

웹 디자인은 결과의 범위가 매우 넓습니다. 훌륭한 디자인은 절제와 발명을 함께 다룹니다. 오랜 시간 검증된 패턴을 바탕으로 하되, 그 안에 새로운 무언가를 더해야 합니다. GPT-5.4는 이런 폭넓은 디자인 스펙트럼을 학습했고, 웹사이트를 구성하는 다양한 방식을 이해합니다.

프롬프트가 충분히 구체적이지 않으면 모델은 학습 데이터에서 자주 등장한 패턴으로 되돌아가는 경향이 있습니다. 그중 일부는 검증된 관습이지만, 상당수는 단지 과도하게 자주 등장했을 뿐인 습관이기도 합니다. 그 결과는 대체로 그럴듯하고 기능적이지만, 구조가 지나치게 평범해지고, 시각적 위계가 약해지며, 머릿속에서 기대한 디자인 수준에 미치지 못하는 선택으로 흘러갈 수 있습니다.

이 가이드는 GPT-5.4가 여러분이 원하는 디자인 방향으로 더 정확히 움직이도록 유도하는 실용적인 기법을 설명합니다.

## 모델 개선 사항

GPT-5.4는 [여러 축에서](https://openai.com/index/introducing-gpt-5-4/) 개선되었지만, 프론트엔드 작업과 관련해서는 특히 다음 세 가지 실질적인 향상에 집중했습니다.

- 디자인 과정 전반에서 더 강력해진 이미지 이해 능력
- 더 완전하고 기능적으로 탄탄한 앱과 웹사이트 생성
- 스스로 작업을 검사하고 테스트하고 검증하기 위한 도구 활용 능력 향상

### 이미지 이해와 도구 활용

GPT-5.4는 이미지 검색 및 이미지 생성 도구를 네이티브하게 사용할 수 있도록 학습되었습니다. 덕분에 시각적 추론을 디자인 과정에 직접 통합할 수 있습니다. 최상의 결과를 원한다면, 최종 에셋을 고르기 전에 먼저 무드보드나 여러 개의 시각적 옵션을 생성하도록 지시하는 것이 좋습니다.

이미지가 담아야 할 속성, 예를 들어 스타일, 색상 팔레트, 구도, 분위기를 명시적으로 설명하면 더 강한 시각적 레퍼런스로 모델을 유도할 수 있습니다. 또한 이미 생성한 이미지를 재사용하라고 하거나, 새 비주얼을 만들기 위해 이미지 생성 도구를 호출하라고 하거나, 필요 시 특정 외부 이미지를 참조하라고 지시하는 프롬프트를 함께 포함하는 것이 좋습니다.

```text
업로드되었거나 미리 생성된 이미지가 있으면 기본적으로 그것을 사용합니다. 그렇지 않다면 이미지 생성 도구를 사용해 시각적으로 뛰어난 이미지를 만듭니다. 사용자가 명시적으로 요청하지 않는 한 웹 이미지를 참조하거나 링크하지 않습니다.
```

### 기능 완성도 향상

이 모델은 더 완전하고 기능적으로 건전한 앱을 개발하도록 학습되었습니다. 따라서 긴 호흡의 작업에서도 더 높은 신뢰성을 기대할 수 있습니다. 이전에는 불가능하다고 느꼈던 게임이나 복잡한 사용자 경험도 이제는 한두 번의 턴 안에서 현실적인 결과로 이어질 수 있습니다.

### Computer Use와 검증

GPT-5.4는 컴퓨터 사용(computer use)을 위해 학습된 OpenAI의 첫 메인라인 모델입니다. 인터페이스를 네이티브하게 탐색할 수 있으며, Playwright 같은 도구와 결합하면 자신의 작업을 반복적으로 점검하고, 동작을 검증하고, 구현을 다듬을 수 있습니다. 그 결과 더 길고 더 자율적인 개발 워크플로우가 가능해집니다.

이 기능이 실제로 어떻게 동작하는지는 [launch video](https://openai.com/index/introducing-gpt-5-4/?video=1170427106%20)에서 확인하실 수 있습니다.

Playwright는 특히 프론트엔드 개발에서 큰 가치를 발휘합니다. 렌더링된 페이지를 검사하고, 여러 viewport를 테스트하고, 앱 흐름을 탐색하고, 상태나 내비게이션 문제를 찾아내는 데 유용합니다. Playwright 도구나 스킬을 제공하면 GPT-5.4가 더 세련되고 기능적으로 완성된 인터페이스를 만들 가능성이 크게 높아집니다. 향상된 이미지 이해 능력 덕분에, 레퍼런스 UI가 주어졌을 때 시각적으로 자신의 작업을 검증하고 실제 결과가 참조 이미지와 일치하는지도 확인할 수 있습니다.

## 실전 팁 빠른 시작

이 문서에서 몇 가지만 가져가고 싶다면, 우선 아래 항목부터 적용해 보세요.

1. 시작은 낮은 추론 수준(low reasoning level)으로 선택합니다.
2. 디자인 시스템과 제약 조건을 미리 정의합니다. 예를 들어 타이포그래피, 색상 팔레트, 레이아웃 등이 있습니다.
3. 스크린샷 같은 시각적 레퍼런스나 무드보드를 제공해 모델이 따를 시각적 가드레일을 마련합니다.
4. 콘텐츠 작성 방향을 잡아 줄 수 있도록 내러티브나 콘텐츠 전략을 먼저 정해 둡니다.

아래는 시작할 때 사용할 수 있는 프롬프트 예시입니다.

```text
## 프론트엔드 작업

프론트엔드 디자인 작업을 할 때는, 평범하고 과하게 조립된 레이아웃을 피합니다.

**다음의 엄격한 규칙을 사용합니다:**
- 하나의 구성: 첫 viewport는 대시보드처럼 보여서는 안 되며, 하나의 완결된 구성처럼 읽혀야 합니다. 단, 실제로 대시보드인 경우는 예외입니다.
- 브랜드 우선: 브랜딩 페이지에서는 브랜드나 제품명이 hero 수준의 신호여야 하며, 단순한 nav 텍스트나 eyebrow로 끝나면 안 됩니다. 어떤 headline도 브랜드보다 더 강하면 안 됩니다.
- 브랜드 테스트: nav를 제거했을 때 첫 viewport가 다른 브랜드의 페이지처럼 보여도 이상하지 않다면, 브랜딩이 너무 약한 것입니다.
- 타이포그래피: 표현력이 있고 목적이 분명한 폰트를 사용하고, 기본 스택(Inter, Roboto, Arial, system)은 피합니다.
- 배경: 단색 배경에만 의존하지 말고, gradient, 이미지, 또는 은은한 패턴으로 분위기를 만듭니다.
- 풀블리드 hero만 사용: 랜딩 페이지와 프로모션 화면에서는 hero 이미지가 기본적으로 화면 끝에서 끝까지 이어지는 지배적인 시각 평면 또는 배경이어야 합니다. 기존 디자인 시스템이 명확히 요구하지 않는 한, inset hero 이미지, 측면 패널형 hero 이미지, 둥근 미디어 카드, 타일형 콜라주, 떠 있는 이미지 블록은 사용하지 않습니다.
- Hero 예산: 첫 viewport에는 보통 브랜드, headline 하나, 짧은 보조 문장 하나, CTA 그룹 하나, 지배적인 이미지 하나만 포함되어야 합니다. 통계, 일정, 이벤트 목록, 주소 블록, 프로모션, "이번 주" 콜아웃, 메타데이터 행, 보조 마케팅 콘텐츠를 첫 viewport에 두지 않습니다.
- Hero 오버레이 금지: hero 미디어 위에 분리된 라벨, 떠 있는 배지, 프로모션 스티커, 정보 칩, 콜아웃 박스를 올리지 않습니다.
- 카드: 기본값은 카드 없음입니다. hero에서는 절대 카드를 사용하지 않습니다. 카드는 사용자 상호작용을 담는 컨테이너일 때만 허용됩니다. 테두리, 그림자, 배경, radius를 제거해도 상호작용이나 이해에 문제가 없다면 그것은 카드가 아니어야 합니다.
- 섹션마다 하나의 역할: 각 섹션은 하나의 목적, 하나의 headline, 그리고 보통 하나의 짧은 보조 문장만 가져야 합니다.
- 실제 시각적 앵커: 이미지는 제품, 장소, 분위기, 맥락을 보여줘야 합니다. 장식용 gradient나 추상 배경은 핵심 비주얼 아이디어로 간주하지 않습니다.
- 군더더기 줄이기: pill 묶음, 통계 띠, 아이콘 행, 박스형 프로모션, 일정 조각, 서로 경쟁하는 여러 텍스트 블록을 피합니다.
- 움직임은 노이즈가 아니라 존재감과 위계를 만드는 데 사용합니다. 시각 중심 작업에서는 최소 2~3개의 의도된 motion을 포함합니다.
- 색과 룩: 분명한 시각 방향을 선택하고 CSS 변수를 정의합니다. 보라색-흰색 기본 조합은 피합니다. 보라색 편향이나 다크 모드 편향도 피합니다.
- 페이지는 데스크톱과 모바일 모두에서 정상적으로 로드되어야 합니다.
- React 코드를 작성할 때는 팀이 이미 사용 중이라면 `useEffectEvent`, `startTransition`, `useDeferredValue` 같은 현대적인 패턴을 우선합니다. 이미 쓰고 있지 않다면 기본적으로 `useMemo`나 `useCallback`을 추가하지 말고, 저장소의 React Compiler 가이드를 따릅니다.

예외: 기존 웹사이트나 디자인 시스템 안에서 작업 중이라면, 이미 확립된 패턴, 구조, 시각 언어를 유지합니다.
```

## 더 나은 디자인을 위한 기법

### 디자인 원칙부터 시작하기

H1 헤드라인은 하나만 둘 것, 섹션은 여섯 개를 넘기지 않을 것, 서체는 최대 두 개까지 사용할 것, 강조 색상은 하나만 둘 것, 첫 화면 위에는 기본 CTA 하나만 둘 것 같은 제약을 먼저 정의해 보세요.

### 시각적 레퍼런스 제공하기

스크린샷이나 무드보드 같은 레퍼런스는 모델이 레이아웃의 리듬, 타이포그래피 스케일, 간격 시스템, 이미지 처리 방식을 추론하는 데 도움을 줍니다. 아래는 GPT-5.4가 사용자의 검토를 위해 직접 무드보드를 생성하는 예시입니다.

![GPT-5.4가 일관된 비주얼 방향을 잡기 위해 생성한 무드보드 예시](https://cdn.openai.com/devhub/blog/codex_moodboard.png)

*NYC 커피 문화와 Y2K 미학에서 영감을 받아 Codex의 GPT-5.4가 만든 무드보드*

### 페이지를 하나의 내러티브로 구성하기

전형적인 마케팅 페이지 구조는 다음과 같습니다.

1. Hero: 정체성과 약속을 확립합니다.
2. Supporting imagery: 맥락이나 환경을 보여줍니다.
3. Product detail: 제공하는 내용을 설명합니다.
4. Social proof: 신뢰를 형성합니다.
5. Final CTA: 관심을 전환 행동으로 이어지게 합니다.

### 디자인 시스템 준수를 지시하기

모델이 빌드 초기에 명확한 디자인 시스템을 세우도록 유도하는 것이 좋습니다. `background`, `surface`, `primary text`, `muted text`, `accent` 같은 핵심 디자인 토큰과, `display`, `headline`, `body`, `caption` 같은 타이포그래피 역할을 정의해 두세요. 이렇게 구조를 잡아 주면 애플리케이션 전반에서 일관되고 확장 가능한 UI 패턴을 만들기 쉬워집니다.

대부분의 웹 프로젝트에서는 **React와 Tailwind** 같은 익숙한 스택으로 시작하는 것이 잘 맞습니다. GPT-5.4는 이런 도구들과 특히 잘 맞기 때문에 빠르게 반복하고 완성도 높은 결과에 도달하기가 수월합니다.

모션과 레이어형 UI 요소는 복잡성을 키울 수 있습니다. 특히 고정 요소나 플로팅 요소가 주요 콘텐츠와 상호작용할 때 더욱 그렇습니다. 애니메이션, 오버레이, 장식 레이어를 다룰 때는 안전한 레이아웃 동작을 유도하는 가이드를 포함하는 것이 좋습니다. 예를 들면 다음과 같습니다.

```text
고정되거나 떠 있는 UI 요소가 어떤 화면 크기에서도 텍스트, 버튼, 기타 핵심 콘텐츠를 가리지 않도록 합니다. 안전한 영역에 배치하고, 적절한 경우에는 주요 콘텐츠 뒤로 보내며, 충분한 간격을 유지합니다.
```

### 추론 수준 낮추기

더 단순한 웹사이트에서는 추론을 많이 한다고 해서 항상 더 좋은 결과가 나오지는 않습니다. 실제로는 **low와 medium reasoning level이 더 강한 프론트엔드 결과로 이어지는 경우가 많습니다.** 모델이 빠르고 집중된 상태를 유지하고 과하게 생각하는 경향을 줄이면서도, 더 대담한 디자인이 필요할 때 추론 수준을 올릴 여지를 남겨 주기 때문입니다.

### 실제 콘텐츠에 디자인을 고정하기

실제 카피, 제품 맥락, 명확한 프로젝트 목표를 제공하는 것은 프론트엔드 결과를 개선하는 가장 간단한 방법 중 하나입니다. 이런 맥락이 있어야 모델이 적절한 사이트 구조를 고르고, 더 분명한 섹션 단위 내러티브를 만들고, 일반적인 플레이스홀더 패턴으로 돌아가지 않으면서 더 그럴듯한 메시지를 작성할 수 있습니다.

## frontend-skill로 전체 흐름 묶기

일반적인 프론트엔드 작업에서 GPT-5.4를 더 잘 활용할 수 있도록, OpenAI는 전용 [`frontend-skill`](https://github.com/openai/skills/tree/main/skills/.curated/frontend-skill)도 준비했습니다. 이 스킬은 구조, 미감, 인터랙션 패턴에 대해 더 강한 가이드를 제공해, 별도 조정이 많지 않아도 더 세련되고 의도적이며 매력적인 디자인을 만들 수 있도록 돕습니다.

```text
---
name: frontend-skill
description: 시각적으로 강한 랜딩 페이지, 웹사이트, 앱, 프로토타입, 데모, 게임 UI를 요구하는 작업에서 사용합니다. 이 스킬은 평범한 카드, 약한 브랜딩, UI 잡음을 피하면서 절제된 구성, 이미지 중심 위계, 응집력 있는 콘텐츠 구조, 세련된 motion을 강제합니다.
---

# 프론트엔드 스킬

작업의 품질이 컴포넌트 개수보다 아트 디렉션, 위계, 절제, 이미지, motion에 더 크게 좌우될 때 이 스킬을 사용합니다.

목표: 의도적이고, 프리미엄하며, 현재적인 감각을 주는 인터페이스를 완성합니다. 기본적으로는 상을 받을 만한 수준의 구성을 지향합니다. 즉, 하나의 큰 아이디어, 강한 이미지, 절제된 카피, 엄격한 spacing, 적은 수의 기억에 남는 motion을 사용합니다.

## 작업 모델

구현 전에 다음 세 가지를 먼저 씁니다:

- visual thesis: 분위기, 재질감, 에너지를 설명하는 한 문장
- content plan: hero, 보조 섹션, 상세 섹션, 최종 CTA
- interaction thesis: 페이지의 인상을 바꾸는 2~3개의 motion 아이디어

각 섹션에는 하나의 역할, 하나의 지배적인 비주얼 아이디어, 하나의 핵심 takeaway 또는 action만 부여합니다.

## 기본 미감 규칙

- 컴포넌트가 아니라 구성부터 시작합니다.
- 풀블리드 hero나 전체 캔버스를 쓰는 시각적 앵커를 선호합니다.
- 브랜드나 제품명을 가장 강하게 보이는 텍스트로 만듭니다.
- 카피는 몇 초 안에 훑어볼 수 있을 만큼 짧게 유지합니다.
- 장식적 chrome를 추가하기 전에 whitespace, alignment, scale, cropping, contrast로 해결합니다.
- 시스템은 제한합니다. 서체는 최대 두 개, 강조 색상은 기본적으로 하나만 사용합니다.
- 기본 레이아웃은 카드 없는 구성을 택합니다. 대신 section, column, divider, list, media block을 사용합니다.
- 첫 viewport는 문서가 아니라 포스터처럼 다룹니다.

## 랜딩 페이지

기본 순서:

1. Hero: 브랜드 또는 제품, 약속, CTA, 지배적인 비주얼 하나
2. Support: 구체적인 기능, 제안, 또는 증거 포인트 하나
3. Detail: 분위기, 워크플로우, 제품의 깊이, 또는 스토리
4. Final CTA: 전환, 시작, 방문, 또는 연락

Hero 규칙:

- 하나의 구성만 사용합니다.
- 풀블리드 이미지 또는 지배적인 시각 평면을 사용합니다.
- 정석적인 풀블리드 규칙: 브랜딩 랜딩 페이지에서는 hero 자체가 페이지 여백, 프레임 컨테이너, 공통 max-width 없이 화면 끝에서 끝까지 이어져야 합니다. 제한은 내부 텍스트/액션 column에만 둡니다.
- 브랜드가 먼저, headline이 두 번째, 본문이 세 번째, CTA가 네 번째입니다.
- 기본적으로 hero 카드, 통계 띠, 로고 클라우드, 과도한 pill 묶음, 떠 있는 대시보드는 두지 않습니다.
- headline은 데스크톱에서는 대략 2~3줄, 모바일에서는 한눈에 읽히는 길이로 유지합니다.
- 텍스트 column은 좁게 유지하고, 이미지 안의 차분한 영역에 고정합니다.
- 이미지 위의 모든 텍스트는 강한 대비와 명확한 탭 타깃을 가져야 합니다.

이미지를 제거해도 첫 viewport가 여전히 성립한다면, 그 이미지는 너무 약한 것입니다. nav를 숨겼을 때 브랜드 존재감이 사라진다면, 위계가 너무 약한 것입니다.

Viewport 예산:

- 첫 화면에 sticky/fixed header가 포함된다면, 그 header도 hero 예산에 포함됩니다. header와 hero 콘텐츠를 합친 전체가 일반적인 데스크톱과 모바일의 초기 viewport 안에 들어와야 합니다.
- `100vh`/`100svh` hero를 사용할 때는 지속적으로 남아 있는 UI chrome 높이를 빼거나(`calc(100svh - header-height)`), header를 일반 흐름에 쌓지 말고 overlay로 처리합니다.

## 앱 UI

기본값은 Linear 스타일의 절제입니다:

- 차분한 surface 위계
- 강한 타이포그래피와 spacing
- 적은 색상 수
- 밀도는 높지만 읽기 쉬운 정보
- 최소한의 chrome
- 카드 자체가 상호작용일 때만 카드 사용

앱 UI는 다음 축을 중심으로 구성합니다:

- 핵심 작업 영역
- 내비게이션
- 보조 컨텍스트 또는 inspector
- action 또는 state를 위한 명확한 강조 색 하나

다음은 피합니다:

- 대시보드 카드 모자이크
- 모든 영역에 두꺼운 테두리 사용
- 일상적인 제품 UI 뒤에 깔리는 장식용 gradient
- 서로 경쟁하는 여러 강조 색
- 훑어보기 경험을 개선하지 못하는 장식용 아이콘

어떤 panel이 카드 처리 없이도 의미를 잃지 않는다면, 카드 스타일을 제거합니다.

## 이미지

이미지는 서사적 역할을 해야 합니다.

- 브랜드, 장소, 에디토리얼 페이지, 라이프스타일 제품에는 최소 하나 이상의 강하고 실제처럼 보이는 이미지를 사용합니다.
- 추상 gradient나 가짜 3D 오브젝트보다 현장감 있는 사진을 선호합니다.
- 텍스트를 올릴 수 있도록 안정적인 톤 영역이 있는 이미지를 고르거나 crop합니다.
- UI와 충돌하는 간판, 로고, 타이포그래피 잡음이 내장된 이미지는 사용하지 않습니다.
- UI 프레임, 분할, 카드, 패널이 내장된 이미지는 생성하지 않습니다.
- 여러 순간을 보여줘야 한다면 하나의 콜라주 대신 여러 이미지를 사용합니다.

첫 viewport에는 실제 비주얼 앵커가 필요합니다. 장식용 텍스처만으로는 부족합니다.

## 카피

- 디자인 해설이 아니라 제품 언어로 씁니다.
- 의미는 headline이 중심적으로 전달하게 합니다.
- 보조 카피는 대체로 짧은 한 문장이면 충분합니다.
- 섹션 간 반복은 줄입니다.
- 프롬프트 문구나 디자인 해설을 UI 안에 넣지 않습니다.
- 모든 섹션에는 하나의 책임만 부여합니다. 설명하거나, 증명하거나, 깊이를 더하거나, 전환시키는 역할입니다.

카피의 30%를 삭제했을 때 페이지가 더 좋아진다면, 더 삭제합니다.

## 제품 UI를 위한 유틸리티 카피

작업 대상이 대시보드, 앱 화면, 관리자 도구, 운영 워크스페이스라면 마케팅 카피보다 유틸리티 카피를 기본값으로 사용합니다.

- 약속, 분위기, 브랜드 보이스보다 방향 안내, 상태, 행동을 우선합니다.
- 작업 화면 자체에서 시작합니다. KPIs, charts, filters, tables, status, task context가 중심이어야 합니다. 사용자가 명시적으로 요청하지 않는 한 hero 섹션을 도입하지 않습니다.
- 섹션 제목은 그 영역이 무엇인지, 또는 사용자가 거기서 무엇을 할 수 있는지 말해 줘야 합니다.
- 좋은 예: "Selected KPIs", "Plan status", "Search metrics", "Top segments", "Last sync".
- 특별히 요청되지 않는 한 제품 화면에서 열망형 hero 문구, 은유, 캠페인식 언어, 경영진 요약 배너는 피합니다.
- 보조 텍스트는 범위, 동작 방식, 최신성, 또는 의사결정 가치를 한 문장으로 설명해야 합니다.
- 어떤 문장이 홈페이지 hero나 광고 문구로도 그대로 들어갈 수 있다면, 제품 UI처럼 들릴 때까지 다시 씁니다.
- 어떤 섹션이 운영, 모니터링, 판단을 돕지 못한다면 제거합니다.
- 판별 질문: 운영자가 heading, label, 숫자만 훑어봐도 페이지를 즉시 이해할 수 있는가?

## Motion

motion은 소음이 아니라 존재감과 위계를 만드는 데 사용합니다.

시각 중심 작업에서는 최소 2~3개의 의도된 motion을 포함합니다:

- hero 안의 entrance sequence 하나
- scroll-linked, sticky, 또는 depth effect 하나
- affordance를 더 선명하게 만드는 hover, reveal, 또는 layout transition 하나

사용 가능하다면 다음에는 Framer Motion을 선호합니다:

- 섹션 reveal
- 공유 layout transition
- scroll-linked opacity, translate, scale 변화
- sticky storytelling
- 공간만 채우는 것이 아니라 서사를 전진시키는 carousel
- menu, drawer, modal의 존재감 효과

Motion 규칙:

- 짧은 녹화에서도 눈에 띄어야 합니다.
- 모바일에서 부드러워야 합니다.
- 빠르되 절제되어야 합니다.
- 페이지 전반에서 일관되어야 합니다.
- 장식용일 뿐이라면 제거합니다.

## 엄격한 규칙

- 기본값은 카드 없음입니다.
- 기본값은 hero 카드 없음입니다.
- 브리프가 풀블리드를 요구할 때는 박스형 또는 중앙 column형 hero를 사용하지 않습니다.
- 섹션당 지배적인 아이디어는 하나만 둡니다.
- 어떤 섹션도 자신을 설명하기 위해 많은 작은 UI 장치에 의존해서는 안 됩니다.
- 브랜딩 페이지에서는 어떤 headline도 브랜드보다 더 강해서는 안 됩니다.
- 의미 없는 filler copy를 넣지 않습니다.
- 텍스트가 차분하고 통일된 한쪽 면에 놓이지 않는 한 split-screen hero를 사용하지 않습니다.
- 명확한 이유 없이 서체를 세 개 이상 쓰지 않습니다.
- 제품에 이미 강한 시스템이 있는 경우가 아니라면 강조 색은 하나만 사용합니다.

## 다음 실패 사례를 거부합니다

- 첫인상이 평범한 SaaS 카드 그리드인 경우
- 이미지는 아름답지만 브랜드 존재감이 약한 경우
- 강한 headline은 있지만 분명한 action이 없는 경우
- 텍스트 뒤에 복잡한 이미지가 깔린 경우
- 같은 분위기 문장을 반복하는 섹션들
- 서사적 목적이 없는 carousel
- 레이아웃 대신 카드 적층으로 만든 앱 UI

## 최종 점검 질문

- 첫 화면에서 브랜드나 제품이 분명하게 드러나는가?
- 강한 시각적 앵커가 하나 존재하는가?
- headline만 훑어봐도 페이지를 이해할 수 있는가?
- 각 섹션이 하나의 역할만 수행하는가?
- 카드가 정말 필요한가?
- motion이 위계나 분위기를 개선하는가?
- 모든 장식용 그림자를 제거해도 여전히 프리미엄한 느낌이 남는가?
```

Codex 앱 안에서는 다음 명령으로 `frontend-skill`을 설치할 수 있습니다.

`frontend-skill`의 도움으로 생성한 웹사이트 예시로는 Landing Pages, Games, Dashboards가 있습니다.
