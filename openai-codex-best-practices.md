# OpenAI Codex - Best Practices

> Getting started with Codex and proven practices for better results

**원문 출처:** [https://developers.openai.com/codex/learn/best-practices](https://developers.openai.com/codex/learn/best-practices)

---

Codex를 처음 사용하시거나 코딩 에이전트 전반에 익숙하지 않으시다면, 이 가이드는 더 빠르게 더 나은 결과를 얻는 데 도움이 됩니다. 이 문서는 CLI, IDE 확장, Codex 앱 전반에 걸쳐 Codex를 더 효과적으로 활용하기 위한 핵심 습관들을 다룹니다. 프롬프팅과 계획 수립부터 검증, MCP, 스킬, 자동화까지 폭넓게 살펴봅니다.

Codex는 일회성 도우미가 아니라 시간을 들여 설정하고 발전시키는 **팀원**처럼 다룰 때 가장 효과적으로 작동합니다. 올바른 작업 컨텍스트로 시작하고, `AGENTS.md`를 통해 지속적인 지침을 제공하며, 워크플로우에 맞게 Codex를 설정하고, MCP로 외부 시스템을 연결하고, 반복 작업을 스킬로 전환하고, 안정적인 워크플로우를 자동화하는 것이 핵심입니다.

## 1. 강력한 첫 사용: 컨텍스트와 프롬프트

Codex는 프롬프트가 완벽하지 않아도 충분히 유용합니다. 최소한의 설정만으로도 어려운 문제를 맡겨 좋은 결과를 얻을 수 있습니다. 명확한 프롬프팅이 반드시 필요한 것은 아니지만, 특히 대규모 코드베이스나 중요한 작업에서는 결과의 신뢰성을 높여줍니다.

대규모 또는 복잡한 저장소에서 작업할 때 가장 큰 효과를 내는 방법은 **올바른 작업 컨텍스트와 명확한 구조**를 Codex에 제공하는 것입니다.

### 프롬프트에 포함해야 할 4가지 요소

| 요소 | 설명 |
|---|---|
| **Goal (목표)** | 무엇을 변경하거나 구축하려고 하는가? |
| **Context (컨텍스트)** | 이 작업에 관련된 파일, 폴더, 문서, 예시, 오류는 무엇인가? `@` 멘션으로 특정 파일을 컨텍스트로 지정할 수 있다. |
| **Constraints (제약 조건)** | Codex가 따라야 할 표준, 아키텍처, 안전 요구사항, 또는 컨벤션은 무엇인가? |
| **Done when (완료 조건)** | 테스트 통과, 동작 변경, 버그 재현 불가 등 작업이 완료되었다고 볼 수 있는 기준은 무엇인가? |

이 네 가지 요소를 포함하면 Codex가 범위를 유지하고, 가정을 줄이며, 검토하기 쉬운 결과물을 만들어냅니다.

### 추론 수준(Reasoning Level) 선택

작업의 난이도에 따라 적절한 추론 수준을 선택하고, 자신의 워크플로우에 맞는 설정을 테스트해 보시는 것이 좋습니다.

| 수준 | 적합한 작업 |
|---|---|
| **Low** | 빠르고 범위가 명확한 작업 |
| **Medium / High** | 복잡한 변경 또는 디버깅 |
| **Extra High** | 장기적이고 에이전트 중심의 추론 집약적 작업 |

> **팁:** Codex 앱에서 음성 받아쓰기(speech dictation)를 활용하면 타이핑 없이도 더 빠르게 컨텍스트를 제공하실 수 있습니다.

## 2. 어려운 작업은 먼저 계획 수립

작업이 복잡하거나 모호하거나 설명하기 어렵다면, 코딩을 시작하기 전에 Codex에게 먼저 계획을 세우도록 요청해 보세요.

### 효과적인 계획 수립 방법

**Plan 모드 사용** — 대부분의 사용자에게 가장 쉽고 효과적인 방법입니다. Plan 모드는 Codex가 컨텍스트를 수집하고, 명확화 질문을 하며, 구현 전에 더 탄탄한 계획을 수립하도록 도와줍니다. `/plan` 또는 `Shift+Tab`으로 전환하실 수 있습니다.

**Codex에게 인터뷰 요청** — 원하는 것에 대한 대략적인 아이디어는 있지만 잘 설명하기 어려운 경우, Codex가 먼저 질문하도록 요청해 보세요. 가정에 도전하고 막연한 아이디어를 코드 작성 전에 구체적인 형태로 전환하도록 안내하면 도움이 됩니다.

**PLANS.md 템플릿 활용** — 더 고급 워크플로우에서는 장기 실행 또는 다단계 작업을 위해 `PLANS.md` 또는 실행 계획 템플릿을 따르도록 Codex를 설정할 수 있습니다.

## 3. AGENTS.md로 지침을 재사용 가능하게

프롬프팅 패턴이 효과적으로 작동하기 시작하면, 다음 단계는 그것을 매번 수동으로 반복하지 않는 것입니다. 바로 이 지점에서 `AGENTS.md`가 필요합니다.

`AGENTS.md`는 **에이전트를 위한 오픈 포맷 README**라고 생각하시면 됩니다. 자동으로 컨텍스트에 로드되며, 저장소에서 Codex가 어떻게 작동하기를 원하는지 팀의 방식을 정리해 두기에 가장 좋은 장소입니다.

### 좋은 AGENTS.md가 다루어야 할 내용

| 항목 | 설명 |
|---|---|
| 저장소 레이아웃 | 중요한 디렉토리 구조 |
| 프로젝트 실행 방법 | 로컬 환경 설정 및 실행 명령 |
| 빌드/테스트/린트 명령 | 자주 사용하는 명령어 목록 |
| 엔지니어링 컨벤션 | PR 기대치 및 코드 스타일 |
| 제약 조건 | 해서는 안 되는 것들(do-not rules) |
| 완료 기준 | 작업이 완료되었다고 볼 수 있는 기준 |

### AGENTS.md 계층 구조

`AGENTS.md` 파일은 여러 수준에서 생성할 수 있습니다. 더 구체적인 파일, 즉 현재 디렉토리에 더 가까운 파일이 우선권을 가집니다.

- **전역(Global):** `~/.codex` — 개인 기본값
- **저장소(Repo) 수준:** 공유 표준
- **하위 디렉토리:** 로컬 규칙

> CLI의 `/init` 슬래시 명령은 현재 디렉토리에 기본 `AGENTS.md`를 스캐폴딩하는 빠른 시작 명령입니다. 좋은 출발점이지만, 팀이 실제로 빌드, 테스트, 리뷰, 배포하는 방식에 맞게 결과를 편집해 두시는 것이 좋습니다.

**실용적으로 유지하는 것이 중요합니다.** 짧고 정확한 `AGENTS.md`가 모호한 규칙으로 가득 찬 긴 파일보다 더 유용합니다. 기본 사항부터 시작하고, 반복되는 실수를 발견한 뒤에만 새 규칙을 추가해 보세요. Codex가 같은 실수를 두 번 반복한다면 회고(retrospective)를 요청하고 `AGENTS.md`를 업데이트하는 방식이 효과적입니다.

## 4. 일관성을 위한 Codex 설정

설정(Configuration)은 세션과 환경 전반에 걸쳐 Codex가 더 일관되게 동작하도록 만드는 주요 방법 중 하나입니다. 모델 선택, 추론 노력, 샌드박스 모드, 승인 정책, 프로필, MCP 설정 등의 기본값을 설정할 수 있습니다.

### 설정 파일 구조

| 위치 | 용도 |
|---|---|
| `~/.codex/config.toml` | 개인 기본값 (Codex 앱에서 Settings → Configuration → Open config.toml로 접근) |
| `.codex/config.toml` | 저장소별 동작 설정 |
| 커맨드라인 오버라이드 | 일회성 상황에만 사용 (CLI 사용 시) |

`config.toml`은 MCP 서버, 프로필, 멀티 에이전트 설정, 실험적 기능 등 지속적인 기본값을 정의하는 곳입니다. 직접 편집하거나 Codex에게 업데이트를 요청하실 수 있습니다.

### 샌드박스 및 승인 설정

Codex는 운영 수준의 샌드박싱을 제공하며, 두 가지 핵심 설정을 제어할 수 있습니다.

- **승인 모드(Approval mode):** Codex가 명령 실행 전 허가를 요청하는 시점을 결정한다.
- **샌드박스 모드(Sandbox mode):** Codex가 디렉토리를 읽거나 쓸 수 있는지, 에이전트가 접근할 수 있는 파일을 결정한다.

코딩 에이전트를 처음 사용하신다면 기본 권한으로 시작하시는 것이 좋습니다. 기본적으로 승인과 샌드박싱을 엄격하게 유지하고, 필요성이 명확해진 뒤에만 신뢰할 수 있는 저장소나 특정 워크플로우에 대해 권한을 완화해 보세요.

> CLI, IDE, Codex 앱은 모두 동일한 설정 레이어를 공유합니다. 많은 품질 문제는 실제로 설정 문제(잘못된 작업 디렉토리, 쓰기 권한 부재, 잘못된 모델 기본값, 누락된 도구 및 커넥터)에서 비롯되므로, 초기에 실제 환경에 맞게 Codex를 설정해 두는 것이 중요합니다.

## 5. 테스트와 리뷰로 신뢰성 향상

Codex에게 변경을 요청하는 것에서 멈추지 않는 것이 좋습니다. 필요할 때 테스트를 생성하고, 관련 검사를 실행하고, 결과를 확인하고, 수락하기 전에 작업을 검토하도록 함께 요청해 보세요.

Codex는 이 루프를 대신 수행할 수 있지만, 무엇이 "좋은 결과"인지에 대한 기준은 알고 있어야 합니다. 그 지침은 프롬프트나 `AGENTS.md`에서 제공할 수 있습니다.

### Codex에게 요청할 수 있는 검증 작업

- 변경 사항에 대한 테스트 작성 또는 업데이트
- 적절한 테스트 스위트 실행
- 린트, 포맷팅, 타입 검사 확인
- 최종 동작이 요청과 일치하는지 확인
- 버그, 회귀, 위험한 패턴에 대한 diff 검토

### 코드 리뷰 옵션

Codex 앱에서는 diff 패널을 전환해 로컬에서 변경 사항을 직접 검토하실 수 있습니다. 특정 행을 클릭하면 다음 Codex 턴의 컨텍스트로 피드백이 전달됩니다.

`/review` 슬래시 명령은 여러 방식의 코드 리뷰를 제공합니다.

| 옵션 | 설명 |
|---|---|
| 베이스 브랜치 대비 리뷰 | PR 스타일의 리뷰 |
| 커밋되지 않은 변경 사항 리뷰 | 스테이징 영역의 변경 사항 검토 |
| 특정 커밋 리뷰 | 개별 커밋 단위 검토 |
| 커스텀 리뷰 지침 | 팀 표준에 맞는 맞춤형 리뷰 |

팀에 `code_review.md` 파일이 있고 `AGENTS.md`에서 이를 참조한다면, Codex는 리뷰 시 해당 지침을 따를 수 있습니다. 이는 저장소와 기여자 전반에 걸쳐 리뷰 동작을 일관되게 유지하려는 팀에 매우 유용한 패턴입니다.

> GitHub Cloud를 사용하신다면 PR에 대한 코드 리뷰를 실행하도록 Codex를 설정할 수 있습니다. OpenAI에서는 Codex가 PR의 100%를 리뷰합니다. 자동 리뷰를 활성화하거나 `@Codex`를 통해 반응형으로 리뷰하도록 설정하실 수 있습니다.

## 6. 외부 컨텍스트를 위한 MCP 활용

Codex에 필요한 컨텍스트가 저장소 외부에 있을 때는 MCP를 활용해 보세요. MCP는 Codex를 이미 사용 중인 도구와 시스템에 연결해, 라이브 정보를 프롬프트에 반복적으로 복사해 붙여 넣을 필요를 줄여줍니다.

**Model Context Protocol(MCP)**은 Codex를 외부 도구와 시스템에 연결하기 위한 오픈 표준입니다.

### MCP를 사용해야 하는 경우

| 상황 | 설명 |
|---|---|
| 컨텍스트가 저장소 외부에 있을 때 | 외부 데이터베이스, 문서, 서비스 등 |
| 데이터가 자주 변경될 때 | 실시간 정보가 필요한 경우 |
| 도구를 사용하게 하고 싶을 때 | 붙여넣은 지침보다 도구 사용이 더 신뢰성 있을 때 |
| 반복 가능한 통합이 필요할 때 | 사용자 또는 프로젝트 전반에 걸친 통합 |

Codex는 OAuth를 사용하는 STDIO 및 Streamable HTTP 서버를 모두 지원합니다. Codex 앱에서 Settings → MCP servers로 이동하면 커스텀 및 추천 서버를 확인하실 수 있습니다. CLI에서는 `codex mcp add` 명령으로 커스텀 서버를 추가할 수 있습니다.

> **권고:** 실제 워크플로우를 열어주는 도구만 추가하시는 것이 좋습니다. 모든 도구를 한꺼번에 연결하기보다는, 이미 자주 수동으로 수행하는 루프를 명확하게 줄여 주는 한두 가지 도구부터 시작해 보세요.

## 7. 반복 작업을 스킬로 전환

워크플로우가 반복적이 되면, 긴 프롬프트나 반복적인 주고받기에 계속 의존하지 않는 편이 좋습니다. **스킬(Skill)**을 사용해 `SKILL.md` 파일에 지침, 컨텍스트, Codex가 일관되게 적용해야 하는 지원 로직을 패키징할 수 있습니다. 스킬은 CLI, IDE 확장, Codex 앱 전반에 걸쳐 작동합니다.

### 좋은 스킬 설계 원칙

각 스킬은 하나의 작업에 집중하도록 범위를 제한하는 것이 좋습니다. 2~3개의 구체적인 사용 사례로 시작하고, 명확한 입력과 출력을 정의하며, 스킬이 무엇을 하고 언제 사용하는지를 설명에 포함해 보세요. 사용자가 실제로 말할 법한 트리거 문구를 함께 넣으면 더 좋습니다.

처음부터 모든 엣지 케이스를 다루려고 하실 필요는 없습니다. 하나의 대표적인 작업으로 시작해 잘 작동하게 만든 다음, 그 워크플로우를 스킬로 전환하고 점진적으로 개선해 나가시면 됩니다.

> **경험 법칙:** 같은 프롬프트를 계속 재사용하거나 같은 워크플로우를 반복해서 수정하고 있다면, 그것은 스킬로 분리할 시점일 가능성이 높습니다.

### 스킬이 특히 유용한 반복 작업

- 로그 분류(Log triage)
- 릴리스 노트 초안 작성
- 체크리스트 기반 PR 리뷰
- 마이그레이션 계획
- 텔레메트리 또는 인시던트 요약
- 표준 디버깅 흐름

### 스킬 저장 위치

| 유형 | 경로 |
|---|---|
| 개인 스킬 | `$HOME/.agents/skills` |
| 팀 공유 스킬 | 저장소 내 `.agents/skills` |

`$skill-creator` 스킬은 스킬의 첫 번째 버전을 스캐폴딩하기에 좋은 출발점이며, `$skill-installer` 스킬을 사용해 로컬에 설치할 수 있습니다. 팀 공유 스킬은 새 팀원 온보딩에도 특히 유용합니다.

---

## 8. 자동화로 반복 작업 처리

워크플로우가 안정화되면, Codex가 백그라운드에서 실행되도록 예약할 수 있습니다. Codex 앱의 **자동화(Automations)** 기능은 반복 작업에 대해 프로젝트, 프롬프트, 주기, 실행 환경을 선택할 수 있게 해줍니다.

Codex 앱의 Automations 탭에서 자동화를 생성할 수 있습니다. 실행할 프로젝트, 실행할 프롬프트(스킬 호출 가능), 실행 주기를 선택할 수 있으며, 자동화가 전용 git 워크트리에서 실행될지 로컬 환경에서 실행될지도 지정할 수 있습니다.

### 자동화에 적합한 작업

- 최근 커밋 요약
- 잠재적 버그 스캔
- 릴리스 노트 초안 작성
- CI 실패 확인
- 스탠드업 요약 생성

> **유용한 규칙:** 스킬은 방법을 정의하고, 자동화는 일정을 정의합니다. 워크플로우에 여전히 많은 조정이 필요하다면 먼저 스킬로 전환해 보세요. 예측 가능해지면 자동화가 훨씬 큰 배율 효과를 발휘합니다.

자동화는 단순한 실행뿐 아니라 **반성과 유지보수**에도 활용할 수 있습니다. 최근 세션을 검토하고, 반복되는 마찰을 요약하며, 시간이 지남에 따라 프롬프트, 지침, 워크플로우 설정을 개선하는 데에도 도움이 됩니다.

---

## 9. 세션 컨트롤로 장기 작업 관리

Codex 세션은 단순한 채팅 기록이 아닙니다. 세션은 시간이 지남에 따라 컨텍스트, 결정, 행동이 축적되는 **작업 스레드**이므로, 잘 관리하면 품질에 큰 영향을 미칩니다.

### 유용한 CLI 슬래시 명령

| 명령 | 설명 |
|---|---|
| `/experimental` | 실험적 기능 전환 및 `config.toml`에 추가 |
| `/resume` | 저장된 대화 재개 |
| `/fork` | 원본 트랜스크립트를 보존하면서 새 스레드 생성 |
| `/compact` | 스레드가 길어질 때 이전 컨텍스트의 요약본 생성 (Codex는 자동으로 대화를 압축하기도 함) |
| `/agent` | 병렬 에이전트 실행 시 활성 에이전트 스레드 전환 |
| `/theme` | 구문 강조 테마 선택 |
| `/apps` | Codex에서 직접 ChatGPT 앱 사용 |
| `/status` | 현재 세션 상태 검사 |

> **원칙:** 하나의 일관된 작업 단위마다 하나의 스레드를 유지하는 편이 좋습니다. 작업이 여전히 같은 문제의 일부라면 같은 스레드에 머무는 것이 더 나은 경우가 많고, 작업이 실제로 분기될 때만 포크하는 것이 좋습니다.

Codex의 멀티 에이전트 워크플로우를 활용해 메인 스레드에서 경계가 명확한 작업을 오프로드할 수 있습니다. 메인 에이전트는 핵심 문제에 집중하고, 탐색, 테스트, 분류 같은 작업에는 서브에이전트를 활용해 보세요.

## 10. 흔한 실수들

Codex를 처음 사용할 때 피하시면 좋은 일반적인 실수들이 몇 가지 있습니다.

| 실수 | 설명 |
|---|---|
| 프롬프트에 지속적인 규칙 과부하 | `AGENTS.md`나 스킬로 옮겨야 할 규칙을 프롬프트에 계속 넣는 것 |
| 에이전트가 작업을 볼 수 없게 하는 것 | 빌드 및 테스트 명령 실행 방법에 대한 세부 정보를 제공하지 않는 것 |
| 다단계 복잡한 작업에서 계획 건너뛰기 | 복잡한 작업에서 계획 수립 단계를 생략하는 것 |
| 워크플로우를 이해하기 전에 전체 권한 부여 | 컴퓨터에 대한 전체 권한을 너무 일찍 부여하는 것 |
| git 워크트리 없이 같은 파일에서 라이브 스레드 실행 | 충돌 위험이 있는 동시 작업 |
| 수동으로 신뢰할 수 없는 작업을 자동화로 전환 | 안정화되지 않은 워크플로우를 자동화하는 것 |
| Codex를 단계별로 지켜봐야 하는 것으로 취급 | 자신의 작업과 병렬로 활용하지 않는 것 |
| 프로젝트당 하나의 스레드 사용 | 작업당 하나의 스레드 대신 프로젝트당 하나의 스레드를 사용하면 컨텍스트가 비대해지고 결과가 나빠짐 |

| 영역 | 핵심 원칙 |
|---|---|
| **프롬프팅** | 목표, 컨텍스트, 제약 조건, 완료 조건의 4요소를 포함하라 |
| **계획 수립** | 복잡한 작업은 Plan 모드나 인터뷰 방식으로 먼저 계획하라 |
| **AGENTS.md** | 반복되는 지침을 `AGENTS.md`에 인코딩하고 실수 후 업데이트하라 |
| **설정** | 개인/저장소 수준의 `config.toml`로 일관된 동작을 보장하라 |
| **테스트 & 리뷰** | 변경 후 테스트, 린트, diff 리뷰까지 Codex에게 요청하라 |
| **MCP** | 외부 컨텍스트가 필요할 때 MCP로 도구를 연결하라 |
| **스킬** | 반복되는 워크플로우는 `SKILL.md`로 패키징하라 |
| **자동화** | 안정적인 워크플로우는 Automations 탭에서 예약하라 |
| **세션 관리** | 작업당 하나의 스레드, 필요 시 포크 또는 멀티 에이전트 활용 |
